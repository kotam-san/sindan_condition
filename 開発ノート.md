### GAS連携でのトラブルシューティング記録

GASへのデータ送信は3段階の試行錯誤があった。次回同様の実装をする際の参考に残す。

| 試行 | 方法 | 結果 | 原因 |
|------|------|------|------|
| 1回目 | POST + `application/json` | `e.postData` が undefined | GASのリダイレクトでボディが消える |
| 2回目 | GET + クエリパラメータ | `e.parameter` が undefined | リダイレクトでクエリパラメータも消える |
| 3回目 | POST + `application/x-www-form-urlencoded` + `mode: "no-cors"` | **成功** | HTMLフォームと同じ形式で CORS プリフライト不要 |

**結論：GAS Web AppへのfetchはPOST + form-encoded + no-corsが確実。**

```js
fetch(GAS_URL, {
  method: "POST",
  mode: "no-cors",
  headers: { "Content-Type": "application/x-www-form-urlencoded" },
  body: new URLSearchParams({ ... }).toString(),
})
```

GAS側では `doPost(e)` で `e.parameter.xxx` から読む。

## 次回同じ構成を作るときのための記録

### 全体デプロイフロー

```
1. GitHubリポジトリ作成（github.com で New repository）
2. ローカルでgit初期化 → push
3. Cloudflare Pages でGitHub連携 → 自動デプロイ設定
4. Cloudflare Worker をwranglerでデプロイ
5. Worker URLをindex.htmlのWORKER_URLに設定 → push → 自動デプロイ
```

### GitへのPATを使ったpush（SSH未設定環境）

SSHキーが設定されていない場合、HTTPS + PATをURLに埋め込む：

```sh
git remote set-url origin https://PAT_TOKEN@github.com/ユーザー名/リポジトリ名.git
git push -u origin main
```

PATはGitHub → Settings → Developer settings → Personal access tokens で発行。

### Cloudflare Pages の設定（単一HTMLファイルの場合）

- Framework preset: **None**
- Build command: **空欄のまま**（ビルド不要）
- Output directory: **`/`**（ルートのまま）
- GitHubのmainブランチにpushするたびに自動デプロイされる

### Cloudflare Worker のデプロイ手順

```sh
npm install -g wrangler       # wranglerインストール
wrangler login                # ブラウザでCloudflareにログイン
cd worker/
wrangler secret put ANTHROPIC_API_KEY  # APIキーをシークレットとして登録（再デプロイ時は不要）
wrangler deploy               # デプロイ → Worker URLが表示される
```

再デプロイ時（コード変更時）は `wrangler deploy` だけでOK。シークレットの再設定は不要。

### セキュリティ設計の考え方

**フロントエンド（ブラウザ）に書いていいもの：**
- `WORKER_URL`（公開前提のエンドポイントURL）
- `GAS_URL`（公開前提のエンドポイントURL）

**フロントエンドに書いてはいけないもの：**
- Anthropic APIキー（悪用されると課金が発生する）
- その他の秘密情報

**解決策：** APIキーはCloudflare Workerのシークレット（サーバー側）に保存する。
ブラウザはWorker URLを叩くだけで、APIキーには触れない。これがWorkerを挟む理由。

### Claude APIプロンプト設計のポイント

**出力形式に `|||` 区切りを使う理由：**
- JSONで返させるとLLMが稀に不正なJSONを出力してパースエラーになる
- Markdownで返させると見出し記号（#, *）が混入して表示が崩れる
- `|||` はプロンプト本文に登場しない記号なので誤検出がない
- `text.split("|||").map(s => s.trim()).filter(Boolean)` でシンプルにパースできる

**6セクション構成が有効な理由：**
単なる「提案3つ」より、「分析 → 問い → アクション → メッセージ」という流れがコーチングの構造に近く、ユーザーの腹落ち感が高い。経営診断Proでも同様の構造が好評だった。

**フォールバック設計：**
- Workerが未デプロイ or エラー時でも、静的テキスト（`SUGGESTIONS`定数）で同じ6セクション表示
- ユーザー体験をAPIエラーで壊さないことが重要
- Worker側にも同様のFALLBACK定数を持つ（二重のフォールバック）

### Gated Content設計のポイント

**なぜこの順番か：**
1. ゲート前で「自分の状態がわかった」という満足感を与える（チャート・ステージ・特徴）
2. 「じゃあどうすればいいの？」が生まれたタイミングでメアドを求める
3. ゲート後に価値の高いコンテンツを出す（AI提案）

メアド入力後はエラーが起きてもゲートを解除する（`setEmailUnlocked(true)` を finally 相当で必ず実行）。
ユーザーをメアド入力フォームで詰まらせてはいけない。
